1
Camera → web pipeline complexity
Use a lightweight tethering app (e.g. digiCamControl on Windows, or custom Python script using gphoto2 on Linux) that watches a folder and POST-uploads each new file to your server via a webhook. This is simpler than a full custom integration and works with most DSLRs and mirrorless cameras.
2
RAW → JPEG conversion is CPU-heavy
Use a serverless cloud function (AWS Lambda or Google Cloud Run) triggered on file upload. Libraries like LibRaw or dcraw handle the conversion. Queue jobs with a Redis/Bull queue so the server never blocks. This scales automatically at events with many cameras.
3
Screenshot prevention is not foolproof
Layer the watermark strategy: (a) display images at a resolution too low to be usable (e.g. 800px wide max), (b) apply a large semi-transparent logo watermark across the center, (c) add CSS pointer-events and user-select:none. The image is useless without purchase — this is the standard industry approach used by Getty, Shutterstock, etc.
4
Real-time sync for live/selection rooms
Use Socket.io (on Node.js) or Supabase Realtime (PostgreSQL-backed). Both handle multi-user rooms well. Supabase is the faster option to build — it provides the database, auth, and realtime in one service with minimal backend code.
5
Storage fills up quickly
Auto-delete temporary files when a live room closes (as you specified). For the permanent gallery, use Cloudinary or AWS S3 with lifecycle policies. Cloudinary also handles image resizing, WebP conversion, and delivery via CDN — cutting both storage cost and page load time significantly.
6
Timeline risk & scope creep
Build in strict phases: launch Phase 1 (portfolio) first — this alone generates business. Add Phase 2 (shop) next. Phases 3 & 4 (live/selection rooms) are advanced features — they can be added after the core business is running. This reduces risk and generates revenue earlier.
